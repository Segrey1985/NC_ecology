# Описание проекта NC_ecology

## 1. Краткое описание

NC_ecology - это программный инструмент для автоматизированной подготовки разделов экологической проектной документации. Программа помогает инженерам-экологам и проектировщикам быстрее формировать раздел ООС на основе уже имеющихся смежных разделов проектной документации.

Основная идея проекта: пользователь загружает архив с PDF-файлами смежных разделов ПД, программа анализирует документы, извлекает из них нужные сведения, формирует структурированные данные и заполняет DOCX-шаблоны глав экологического раздела.

Результат работы программы - архив с заполненными документами Word, JSON-файлами с извлеченными данными и служебными логами. Эти материалы предназначены для дальнейшей проверки и доработки специалистом.

## 2. Для кого предназначена программа

Программа ориентирована на пользователей, которые участвуют в подготовке экологической проектной документации:

- инженеров-экологов;
- проектировщиков;
- специалистов по разделу ООС;
- сотрудников проектных организаций;
- технических специалистов, которые интегрируют генерацию документов в существующий рабочий процесс.

Для конечного пользователя программа решает практическую задачу: уменьшает объем ручного поиска информации в смежных разделах и ускоряет подготовку типовых фрагментов экологической документации.

## 3. Какую проблему решает проект

При подготовке раздела ООС специалисту часто нужно вручную просматривать большое количество смежных разделов проектной документации: пояснительную записку, схему планировочной организации земельного участка, архитектурные решения, инженерные изыскания, разделы по инженерным системам и другие материалы.

Из этих документов требуется перенести в экологический раздел сведения об объекте, территории, инженерных условиях, геологии, гидрогеологии, земельных ресурсах и других характеристиках. Такой процесс занимает много времени, зависит от внимательности исполнителя и может приводить к пропускам или несогласованности формулировок.

NC_ecology автоматизирует этот процесс: программа ищет релевантные фрагменты в исходных PDF, извлекает данные с помощью языковой модели, структурирует результат и подставляет его в шаблон документа.

## 4. Основной пользовательский сценарий

Типовой сценарий работы выглядит так:

1. Пользователь собирает смежные разделы проектной документации в ZIP-архив.
2. Внутри архива находятся PDF-файлы с корректными именами, например `1_...`, `2_...`, `3_...`, `5.1_...`, `6.1_...`, `ИГИ_...`.
3. Пользователь отправляет архив в API программы.
4. Программа распаковывает архив, проверяет входные файлы и индексирует PDF-документы.
5. Для каждой главы запускается поиск нужной информации по исходным документам.
6. Языковая модель формирует ответы в структурированном виде.
7. Программа собирает данные в JSON и заполняет DOCX-шаблоны.
8. Пользователь получает ZIP-архив с результатами.

Главный пользовательский результат - глава или перечень глав раздела ПМООС в формате word, которые можно открыть в Microsoft Word, проверить, отредактировать и использовать как основу для дальнейшей работы.

## 5. Какие главы формирует программа

На текущем этапе проект формирует три блока экологической документации.

### Глава 0. Аннотация и введение

Этот блок содержит базовые сведения, которые нужны для начала документа: аннотацию, вводные формулировки и общие проектные данные. Часть значений может заполняться из заранее подготовленных плейсхолдеров и табличных данных.

### Глава 1. Общие сведения об объекте проектирования

Глава посвящена общей характеристике объекта. В нее могут попадать сведения из пояснительной записки, схемы планировочной организации земельного участка, архитектурных решений и других смежных разделов.

### Глава 2. Воздействие объекта на земельные ресурсы

Глава описывает вопросы, связанные с земельными ресурсами, инженерно-геологическими условиями, гидрогеологией, характеристиками территории и другими данными, которые могут быть важны для оценки воздействия объекта.

## 6. Что пользователь загружает на вход

Основной входной файл - ZIP-архив со смежными разделами проектной документации в формате PDF.

Важные требования к входным данным:

- внутри архива должен быть хотя бы один PDF-файл;
- архив не должен быть пустым или поврежденным;
- зашифрованные ZIP-архивы не поддерживаются;
- размер ZIP-архива ограничен 200 МБ;
- суммарный распакованный размер ограничен 800 МБ;
- количество файлов внутри архива ограничено 5000;
- имена PDF-файлов должны начинаться с кода раздела + символ `_` (см. пункт 4)

Примеры ожидаемых префиксов в именах файлов:

- `1_` - пояснительная записка;
- `2_` - схема планировочной организации земельного участка;
- `3_` - архитектурные решения;
- `4_` - конструктивные решения;
- `5_`, `5.1_`, `5.2_`, `5.3_`, `5.4_`, `5.5_`, `5.6_` - инженерные системы;
- `6_`, `6.1_`, `6.2_` - технологические решения;
- `7_` - проект организации строительства;
- `8_` - охрана окружающей среды;
- `9_`, `9.1_`, `9.2_` - пожарная безопасность;
- `10_` - требования к обеспечению безопасной эксплуатации;
- `ИГИ_` - инженерно-геологические изыскания;
- `ИЭИ_` - инженерно-экологические изыскания.

Корректное именование файлов важно, потому что программа использует эти префиксы для понимания, к какому разделу относится документ.

## 7. Что пользователь получает на выходе

После обработки пользователь получает ZIP-архив с результатами. В зависимости от выбранного эндпоинта архив может содержать материалы по одной главе или сразу по всем доступным главам.

В составе результата могут быть:

- заполненные DOCX-документы;
- JSON-файлы с извлеченными и структурированными данными;
- служебные логи выполнения;
- промежуточные данные, полезные для проверки качества генерации.

DOCX-файлы можно открыть в Word и использовать как основу для дальнейшей редакторской и экспертной доработки. JSON-файлы помогают понять, какие данные были извлечены и подставлены в шаблон.

## 8. Как работает программа простыми словами

Программа использует подход RAG: сначала ищет нужную информацию в документах пользователя, а затем передает найденный контекст языковой модели для формирования ответа.

Упрощенно процесс состоит из нескольких этапов:

1. PDF-документы разбиваются на текстовые фрагменты.
2. Фрагменты индексируются в векторной базе данных Qdrant.
3. Для каждого вопроса или поля шаблона программа ищет наиболее подходящие фрагменты исходных документов.
4. Найденные фрагменты дополнительно ранжируются, чтобы повысить качество контекста.
5. Языковая модель формирует ответ в заданной структуре.
6. Ответ проверяется и приводится к ожидаемой JSON-структуре.
7. Итоговые данные передаются в DOCX-шаблон.
8. На выходе формируется готовый архив с документами.

Такой подход позволяет программе опираться не на абстрактные знания модели, а на конкретные документы, загруженные пользователем.

## 9. Почему это полезно пользователю

Главная ценность NC_ecology - сокращение времени на подготовку черновиков экологической документации.

Пользователь получает следующие преимущества:

- меньше ручного поиска по PDF-файлам;
- быстрее формируются повторяющиеся разделы документации;
- сведения берутся из смежных разделов одного и того же проекта;
- результат сохраняется в привычном формате DOCX;
- структурированные JSON-файлы позволяют проверять, какие данные были использованы;
- можно запускать генерацию одной главы или всех доступных глав сразу;
- программа подходит для встраивания в рабочий процесс через API.

## 10. Важные ограничения

NC_ecology не заменяет эксперта и не выпускает финальный нормативный документ без проверки человеком.

Пользователю важно понимать ограничения:

- сгенерированный текст нужно проверять и редактировать;
- качество результата зависит от качества исходных PDF;
- если в исходных документах нет нужной информации, программа не сможет надежно ее восстановить;
- некорректные имена файлов могут ухудшить поиск по разделам;
- табличные данные могут требовать отдельной ручной подготовки;
- на текущем этапе основной поддерживаемый формат исходных документов - PDF;
- Word-файлы как входной формат находятся в планах развития.

Если в шаблоне есть поле, для которого не удалось найти значение, программа может оставить специальную пометку вида `{{undefined ...}}`. Это помогает пользователю увидеть незаполненные места и доработать документ вручную.

## 11. Доступные API-эндпоинты

Программа предоставляет REST API.

Основные эндпоинты:

- `GET /health` - проверка доступности сервиса;
- `POST /chapter0` - генерация аннотации и введения;
- `POST /chapter1` - генерация главы 1;
- `POST /chapter2` - генерация главы 2;
- `POST /chapters/all` - генерация всех доступных глав одним запросом.

Для пользователя наиболее удобный сценарий - `/chapters/all`, потому что он позволяет одним запросом получить весь комплект доступных результатов.

## 12. Инфраструктурные требования

Для работы программы нужны:

- запущенный API-сервис NC_ecology;
- векторная база данных Qdrant;
- доступ к языковой модели через настроенный API-ключ;
- доступ к модели эмбеддингов и reranker-модели;
- корректно настроенный файл окружения `.env`.

В проекте предусмотрен запуск через Docker Compose. Такой способ удобен для развертывания сервиса и Qdrant вместе.
